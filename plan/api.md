# 마켓플레이스 API 요청서 + 동기화 설계

> **작성일**: 2026-03-17
> **목적**: 마켓 개발자에게 전달할 API 스펙 요청 + 우리 쪽 동기화 설계

---

## Part 1: 마켓 개발자에게 요청할 API

### 필수: 판매된 코드 목록 조회

마켓에서 판매 완료된 코드 목록을 조회하는 API가 필요합니다.

#### 엔드포인트 (예시)

```
GET /api/sold-codes
```

#### 인증

| 방식 | 설명 |
|------|------|
| API Key (헤더) | `Authorization: Bearer <API_KEY>` |

> 다른 인증 방식(OAuth 등)도 가능하지만, API Key가 가장 단순합니다.

#### 요청 파라미터

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|------|------|
| `page` | number | N | 페이지 번호 (기본 1) |
| `per_page` | number | N | 페이지당 개수 (기본 100, 최대 1000) |

#### 응답 형식

```json
{
  "data": [
    {
      "code": "GO-000123-A1B2C3",
      "sold_at": "2026-03-17T10:30:00Z"
    }
  ],
  "pagination": {
    "page": 1,
    "per_page": 100,
    "total": 847
  }
}
```

- `data[].code` — 판매된 코드값 (우리가 등록한 코드와 동일한 값)
- `data[].sold_at` — 판매 시각 (ISO 8601)
- `pagination.total` — 전체 판매 건수 (페이지네이션 계산용)

---

### 선택 (있으면 좋음): `since` 날짜 필터

데이터가 많아지면 매번 전체를 조회하는 건 비효율적입니다.
**특정 시점 이후 판매분만** 조회하는 필터가 있으면 좋습니다.

```
GET /api/sold-codes?since=2026-03-17T10:00:00Z&page=1&per_page=1000
```

| 파라미터 | 타입 | 설명 |
|----------|------|------|
| `since` | ISO 8601 datetime | 이 시점 이후에 판매된 코드만 반환 |

> 초기에는 없어도 되지만, 코드 수가 수천~수만 건으로 늘어나면 필수입니다.

## Part 2: 우리 쪽 동기화 설계

### 폴링 방식

Vercel + Supabase 무료 플랜을 사용 중이라 서버 사이드 Cron은 불가합니다.

| 방식 | 설명 |
|------|------|
| **브라우저 폴링** | 대시보드가 열려 있을 때 설정된 주기(5/10/30분)마다 자동 동기화 |
| **수동 버튼** | 설정 페이지에서 "지금 동기화" 클릭 |

### 동기화 로직

```
1. sync_logs에 'running' 레코드 생성
2. 마지막 성공 동기화의 started_at 조회 → since 값으로 사용
   - 첫 실행이면 since 없이 전체 조회
3. 마켓 API 호출 (페이지네이션으로 전체 수집)
   - per_page=1000, 모든 페이지 순회
4. 우리 DB와 비교
   - 마켓에서 sold인데 우리 DB에서 registered → sold로 전환
   - 마켓에 있는데 우리 DB에 없는 코드 → not_found 기록
5. sync_logs 업데이트 (success/failed, 카운트, 상세 내역)
```

### since 증분 조회

- 마지막 성공 동기화의 `started_at`을 `since` 파라미터로 전달
- 마켓 API가 `since`를 지원하지 않으면 파라미터 무시하고 전체 조회
- 전체 조회해도 DB 비교로 중복 처리되므로 정합성 문제 없음

### 에러/실패 처리

| 상황 | 처리 |
|------|------|
| API 호출 실패 (네트워크/5xx) | sync_logs에 `failed` 기록, 다음 주기에 재시도 |
| API 인증 실패 (401/403) | `failed` 기록, 설정 확인 필요 |
| 코드 매칭 불가 (DB에 없음) | `not_found` 카운트 증가, details에 목록 기록 |
| 동기화 중 중복 실행 | 이미 running인 sync가 있으면 스킵 (409 응답) |

## Part 3: 논의 필요 사항

마켓 개발자와 확인이 필요한 항목들:

- [ ] **API 인증 방식 확정** — API Key? OAuth? 다른 방식?
- [ ] **응답 필드 확정** — `code`, `sold_at` 외에 추가 필드가 있는지 (구매자 정보, 주문 ID 등)
- [ ] **`since` 필터 지원 여부** — 지원 가능하면 초기부터 구현 요청
- [ ] **Rate limit 정책** — 분당/시간당 최대 호출 횟수
- [ ] **`per_page` 최대값** — 1000까지 가능한지
- [ ] **전체 목록 vs 변경분만** — 매번 전체를 줄 수 있는지, 변경분만 제공 가능한지
- [ ] **엔드포인트 URL 확정** — 실제 도메인 및 경로
